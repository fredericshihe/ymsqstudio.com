# 琴房管理系统 全面排查报告（性能 / 同步 / 联网 / 流畅度 / 实时性 / 多设备并发）

- 日期：2026-07-08
- 范围：`index.html`（约 818KB，单文件内联脚本），Supabase 后端（表：rooms / room_database / student_database / student_time_slots / practice_logs / practice_alerts / student_schedules），GitHub Pages（main 根目录 → practiceymsq.com）
- 方法：5 路并行代码审计（并发/实时、联网、渲染流畅度、持久化/数据完整性、功能正确性）+ 对最高严重级问题做后端/代码实证核对
- 说明：本报告只做**排查与定位**，未改任何代码。标 ✅ 者为本次已实证确认；其余为审计定位、附 `index.html:行号`。

## 一、结论概览（按优先级）

| 级别 | 数量 | 含义 |
|---|---|---|
| **P0 紧急** | 2 | 数据丢失 / 两位老师静默互相覆盖 |
| **P1 重要** | 8 | 正常使用下出错、状态不一致或不自动恢复 |
| **P2 中等** | 12 | 边界竞态、浪费请求、可扩展性/安全隐患 |
| **P3 优化** | 8 | 加固、打磨、清理 |

**最该先修的 3 件事**：
1. 多人登记（多学生）写库**没有版本校验** → 并发下静默覆盖（P0-1）
2. 练习记录只存在内存里，**刷新/关页即丢**（P0-2）
3. 学生**改名不级联** → 时段/练习记录/提醒全部对不上（P1-3）

**整体评价**：核心的单人 assign/clear/transfer 走的是数据库 CAS（`version` 乐观锁），渲染层有 rAF 批处理 + 精细 DOM patch + 拼音索引缓存，联网稳态请求量不高（约 140 请求/小时/设备）。真正的风险集中在**多人登记旁路了 CAS**、**练习记录持久化**、**改名级联**、以及**两套学生存储并存**这几处。

## P0 — 紧急（数据丢失 / 静默覆盖）

### ✅ P0-1 多人登记旁路 CAS，并发下静默覆盖别人的登记
- 位置：`index.html:9695`（`assignMultipleStudentsToRoom`，`.upsert({... version: nextVersion ...})`）
- 现象/影响：老师 A 用「多人登记」把学生写入房间 X 时，若老师 B 刚刚（单人登记/转移/也在多人登记）占用了 X，A 的写入会**直接覆盖 B 的占用，无报错、无冲突提示**，B 的学生被挤掉。
- 根因：单人 assign/clear/transfer 走 `executeConditionalRoomCloudMutation`，用 `.update(...).eq('version', 期望版本)` 由数据库强制 CAS（`index.html:11376`）；但多人登记用的是裸 `.upsert()`，只是把 `version` 当普通字段写进去，没有 `.eq('version')` 条件。第 9633–9655 的「预检查」是先读后写，中间任何写入都会被覆盖（TOCTOU）。也没有 `runSerializedRoomOperation` 本地串行化。
- 建议修复：把多人登记改走 `executeConditionalRoomCloudMutation`（`.update().eq('version', baseVersion)`），命中 `VERSION_MISMATCH` 时按单人路径处理（回滚 + `syncRoomFromRemoteRow` + `scheduleRoomStateReconcile`）；入口用 `runSerializedRoomOperation([roomName], 'assign-multi', …)` 包一层。

### ✅ P0-2 练习记录仅存内存，刷新/关页/崩溃即永久丢失
- 位置：`index.html:4828`（`localPracticeLogs: []`），写入点 `9725`、`17081`，读取 `12890`
- 现象/影响：每次 assign/clear 都往 `Store.localPracticeLogs` 推一条 `synced:false`。若当下 Supabase 写入失败（离线/RLS/网络），这条记录**只在内存**。刷新、关标签、崩溃在下次成功同步前发生 → 该练习记录**静默丢失**。离线使用时必然丢。
- 根因：练习记录队列没有任何 localStorage 落盘（对比学生的待同步队列 `yms_students_pending_ops_v1` 是持久化的）。
- 建议修复：把 `localPracticeLogs`（至少 `synced:false` 子集）在 push 时与 `beforeunload` 时写入一个带版本号的 LS key，启动时读回并补传。

## P1 — 重要（正常使用下出错 / 不一致 / 不恢复）

### ✅ P1-3 `syncPracticeLogsToCloud` 手动补传永远失败
- 位置：`index.html:12898`（`.insert(unsyncedLogs)`）
- 现象/影响：`unsyncedLogs` 直接把日志对象原样插入，仍带着本地字段 `synced:false`（可能还有 `piano_type/location/...` 不存在的列）。`practice_logs` 没有 `synced` 列 → 报 42703，且**没有像 `writePracticeLog` 那样逐列剔除重试**（对比 `17053` 有 `delete currentData.synced`）。结果「同步到云端」补传练习记录这条路 100% 失败，日志继续留 `synced:false`。
- 建议修复：插入前 `delete log.synced`（并剔除未知列），复用 `writePracticeLog` 的 42703 降级重试逻辑；逐条确认成功后再置 `synced:true`。

### P1-4 多人登记失败回退会重复写练习记录
- 位置：`index.html:9724-9762`
- 现象/影响：多人登记先在 `9725` 把所有记录推入 `localPracticeLogs`（`synced:false`）；若批量 insert 失败进入逐个回退（`9758`），每个 `writePracticeLog` 又推**第二条**并再插云端。结果：内存重复 + 原 `9725` 那批仍 `synced:false`，日后 `syncPracticeLogsToCloud` 再插一次 → `practice_logs` 出现重复 `assign` 行（表无唯一约束，无法去重）。
- 建议修复：回退前先移除/标记预推的那批，或批量 insert 成功后再推本地缓存。

### P1-5 学生改名不级联，时段/记录/提醒全部对不上
- 位置：`index.html:18345-18353`（编辑弹窗 esSave，改名分支只 `update(student_database).eq('name', oldName)`）
- 现象/影响：`student_time_slots`、`student_schedules`、`practice_logs`、以及**当前占用房间里的占用者姓名**都以 `student_name` 为事实外键。改名只改了 `student_database` 一张表 → 学生的练琴时段全部「丢失」、时段提醒不再触发、CSV 导出/时段查询错位、历史记录挂在旧名下。（对比：删除学生**有**级联删 `student_time_slots`，见 `18806`。）
- 建议修复：改名时一并 `update ... set student_name=newName where student_name=oldName` 更新 `student_time_slots / student_schedules / practice_logs`，并改写在住房间的占用者字符串；最好服务端事务化。

### P1-6 新增琴房失败无回滚，可能卡死 / 空客户端 TypeError
- 位置：`index.html:16575-16624`（`createNewRoom`）
- 现象/影响：房间先 push 进 `Store.rooms/byName`（16575），再 `await ...upsert()`。upsert 抛错时 catch 只 toast「添加失败」，**不回滚本地、不关弹窗**：① 出现一个云端不存在的幽灵房间，刷新即消失；② 重试同名被「房间名称已存在」挡住，需刷新才能再建；③ 离线判断 `if(!Store.settings.offlineMode)` 少了别处都有的 `&& supabaseReady`，离线且未置 flag 时会对 null 客户端调用 → 必定 TypeError → 卡死。
- 建议修复：先云端写、成功再改本地，或 catch 里回滚 `Store.rooms/byName/floorSet`；补 `&& supabaseReady`；失败关弹窗并给真实错误。

### P1-7 rooms 实时断线重连后无全量补拉，房间网格会滞留旧态
- 位置：`index.html:11835-11848`（rooms 订阅 `subscribe` 回调）对比 `11888-11891`（students 路径）
- 现象/影响：页面可见时 WebSocket 短暂断开（CLOSED/TIMED_OUT → 重订阅），断开期间对 `rooms` 的增删改**不会补发**，网格显示旧态（幽灵占用/漏掉的清空/漏掉的新房间），直到偶发自愈。
- 根因：rooms 重连只 toast「实时同步已恢复」，不触发重拉；students 路径会 `scheduleStudentReconnectSync`。全量房间重同步只在页面隐藏 > 5 分钟时跑（`3661`），后台 `reconcileLoop` 每 60s 只抽查 5 个房间（`8889`），大房间集收敛很慢。
- 建议修复：rooms `subscribe` 回调里，`hadReconnectAttempts` 时调用 `fetchRooms()`/差异重对账，对齐 students 路径。

### P1-8 房间转移非原子，存在「学生不在任何房间」窗口
- 位置：`index.html:10053-10095`（`transferStudent`）
- 现象/影响：A→B 转移先清源房间（10058），sleep 200ms（10092），再占目标（10095）。这段窗口里其他设备看到学生**哪个房间都不在**。若占 B 失败（B 被并发占用），回滚重占源房间（10111）；若回滚也输了 CAS，学生**落到没有房间**（有错误 toast，但状态已丢）。
- 根因：无两行原子操作；`runSerializedRoomOperation` 只在**本机**串行，挡不住别的设备在窗口里抢 B；200ms 是固定猜测值。
- 建议修复：改为「先占目标再清源」（学生永不落空），或服务端事务原子转移（一次更新两行 + 版本校验）；回滚失败时把学生钉回源房间并强制对账。

### P1-9 学生选择弹窗搜索每键全量重排重建，无防抖 → 卡顿
- 位置：`index.html:16249`（`studentSearch` input 直接绑 `renderStudentList`）/ `16071-16202`
- 现象/影响：房间登记的学生选择框里每敲一个字都同步全量：`matchPinyin` 过滤全体 + 建全房间 `occupancyMap` + **整表 localeCompare 重排** + `box.replaceChildren()` 全量重建每一行（无 diff）。学生多时逐字卡顿，平板尤甚。旁边的学生库面板反而做对了（150ms 防抖 + `patchStudentList` 增量）。
- 建议修复：input 加 ~120-150ms 防抖；或复用 `patchStudentList` 增量 diff；排序/occupancyMap 可在弹窗会话内记忆化，移出每键路径。

### P1-10 RequestThrottler 对所有写路径是死代码
- 位置：`index.html:2432-2438`（`minimumIntervals`），`canExecute` 实际只在 `reconcile`(10672)/`heartbeat`(11972，已禁用) 调用
- 现象/影响：定义了 `roomUpdate/studentUpdate/syncToCloud` 的最小间隔，但**没有任何写路径真正调用**。快速连点或某个 bug 循环可以零限流地猛打 `rooms.upsert`/`practice_logs.insert`。目前靠「每房间操作锁 + 乐观 UI」兜着，属潜在而非正在发生。
- 建议修复：在 `assignRoomUnified/clearRoomUnified/syncRoomToCloud` 接入 `canExecute('roomUpdate', roomName)`；或删掉误导性的未用配置。

## P2 — 中等（边界竞态 / 浪费 / 可扩展性 / 安全）

### P2-11 两套学生存储并存，可能分叉 / 重复插入
- 位置：`Store.students`（`fetchStudents`，`syncStudentDatabaseToCloud` 12724）vs `StudentLibrary.store.items`（`flushPending` 18112）
- 现象/影响：两者都对**无唯一约束**的 `student_database` 做「按名不存在则插入」。手动全量同步与 pending flush 若并发，都判「名字不存在」各插一次 → 重复行。且 `Store.students` 只在全量拉取时刷新，与实时变动的 StudentLibrary 会漂移，是「学生列表与学生库不一致」的根源。渲染层已优先用 StudentLibrary（`16079` 回退才用 `Store.students`）。
- 建议修复：以 StudentLibrary 为唯一真相，`Store.students` 只读派生；长期给 name 加唯一索引后用真 upsert。

### P2-12 loadFromCloud 与 reconcile 删除策略不一致 + 无分页（>1000 行会误删本地）
- 位置：`loadFromCloud`（只合并不删，18019）vs `reconcileStudentsWithCloud`（删本地云端没有的，18451）；相关 select 均无 `.range()`（`5452/18010/18441/18898`）
- 现象/影响：同一「别的设备删了某学生」，看跑到哪条路，本地要么残留要么被删，不一致。**放大风险**：这些 select 不分页，超过 Supabase 默认 1000 行上限后，reconcile 会把第 1000 行之后的本地学生**全部删掉**。当前 214 名学生未触发，但学生数一旦破千即升级为 P1 数据丢失。
- 建议修复：统一一套对账（软删/墓碑语义）；批量本地删除前加 `.range()` 分页或行数守卫。

### P2-13 房间卡片 HTML 未转义房间名，破坏卡片操作 + XSS
- 位置：`index.html:15583, 15623, 15634`（`roomCardHTML` 里 `data-room="${r.name}"`、`title="${r.name}..."` 未 `escapeHtml`，而相邻 `location/remark` 有转义）
- 现象/影响：房间名含 `"` 或 `<` 会破坏卡片属性；而 clear/renew/assign 都靠 `card.dataset.room` → `Store.byName.get(name)` 定位，`data-room` 一坏，这张卡的所有操作静默失效，同时是属性注入/XSS 面。房间名用户可输（新建/内联编辑/CSV 导入）。
- 建议修复：所有 `${r.name}` 属性插值套 `escapeHtml`。

### P2-14 顶部全局搜索无防抖，每键全语料扫描 + innerHTML 重建
- 位置：`index.html:16627` / `16628-16670`（`runGlobalSearch`）。建议 ~150ms 防抖（结果已限 30 条，DOM 不大，省的是每键全量扫描+重建）。

### P2-15 房间网格虚拟渲染在 https 下实际关闭
- 位置：`index.html:14152-14161`（`getRoomGridVirtualRenderConfig` 只在 `file:` 协议启用）。房间数多（60+）时正式站点走全量 `grid.innerHTML=` 一次性主线程解析，结构性变更（首屏/增删/筛选）会有可见卡顿。建议放开 http(s) 限制、保留 60 房间阈值。

### P2-16 客户端时钟偏差导致时长显示错误（serverTimeSync 是空操作）
- 位置：`index.html:2519-2594`（`ServerTimeSync` 硬编码 `offset=0`）。`register_time` 由执行 assign 的设备用本机时间戳，另一台设备用自己的时钟算时长；两机时钟差几分钟则「已练习 N 分钟」错误甚至为负。冲突判定是版本制（不受影响），这是**显示/时长**问题。建议用数据库 `now()`（RPC）算真实 offset，或时长以服务端相对时间渲染。

### P2-17 serverTimeSync/manualSync 的连通性请求纯属浪费
- 位置：`index.html:2546-2568, 15672`。每 5 分钟 + 每次切回标签都 `rooms.select limit 1` 只为「测连通」，offset 永远 0。建议移除或加节流/「距上次 > N 分钟」判断，避免频繁切标签刷请求。

### P2-18 reconcileLoop 定时器不受实时健康度门控
- 位置：`index.html:8885-8914`。固定 60s 递归，只要 `supabaseReady && Store.rooms.length` 就拉取，是稳态最大请求源（~60/小时/设备）。建议实时健康时（`Date.now()-lastActivity < LOOP_INTERVAL`）跳过拉取，稳态请求可减半。

### P2-19 实时事件合并会吞掉中间态（清空→再占的过渡 toast/副作用）
- 位置：`index.html:2339-2367, 2369-2393`。48ms 窗口内同房间两事件合并只留最新；快速 clear→重占会塌缩成一个 UPDATE，「释放」toast 与 occupied→empty 全量重渲染分支不触发。最终状态对，过渡副作用丢。若练习日志依赖实时事件驱动需关注。

### P2-20 `pendingAssign.version >= data.version` 过于激进，丢掉等版本的合法远端写
- 位置：`index.html:12429-12433`。A 乐观占用到 N+1 但输给 B（B 写了不同学生也是 N+1），B 的实时事件被 A 以 `>=` 丢弃，A 短暂显示错误学生直到自己失败的 CAS 返回或对账。建议只在「本地已确认提交」时跳过，或比对占用者身份不同就放行。

### P2-21 localStorage 配额溢出被静默吞掉
- 位置：`index.html:17971/17973/18055/18224`（`save/savePending` 的 catch 只 warn）。配额满时保存静默失败，用户以为存了、刷新即丢（含本应跨刷新存活的 pending 学生操作）。且按天的 `practice_alerts_silence_cache_<day>` 只清「今天」，历史累积。建议专门识别 `QuotaExceededError` 并 toast + 主动清理过期按天缓存。

### P2-22 多人登记确认失败/取消后残留状态
- 位置：`index.html:16413-16441`（`performMultiStudentAssign` 只在 `result.ok` 清理多选态）。cancelled/error 时留在多选模式、选择不清、确认按钮计数不重置，需手动清。建议 `finally`/错误分支重置。

### P2-23 fetchRoomUsageCounts 每次开弹窗都查、无缓存（本次新增功能）
- 位置：`index.html:16003-16021`（本会话加的「本房常练学生」排序）。反复开关同一房间会重复查同一 7 天窗口。建议给 `__roomUsageCounts[roomName]` 加短 TTL（如 60s）缓存。

## P3 — 优化 / 加固 / 清理

- **P3-24** `addStudent` 去重用未 trim 的名、却存 trim 后的名（`18199` vs `18215`）：`"李明 "` 能绕过对 `"李明"` 的查重造出近似重复。统一先 trim 再查重。
- **P3-25** 练习时长 `Date.now()-startMs-60000` 写死减 60s 准备时间，无法反映 renew/transfer 重置；无匹配 assign 的 clear 不写任何时长（`17182-17189`）。建议按配对的 assign 日志 `session_start` 算，准备时间记为数据而非魔数。
- **P3-26** 每张 `.room-card` 常驻 `will-change:transform`（`231`）+ 多个 `infinite` box-shadow 关键帧（`253/275/517-519/744`）：图层膨胀 + 持续重绘。建议只在动画卡片上临时加 will-change，脉冲改用 transform/opacity。
- **P3-27** ~1600 行测试/基准harness（`~6403-8000`）随生产下发，仅 `file:` + URL 参数启用，正式用户是死代码但撑大 818KB 首屏。建议构建时剥离到 dev 文件。
- **P3-28** 新增琴房弹窗每次开都加 backdrop-click 监听、只在 backdrop 关时移除（`16467-16472`），监听器累积。建议一次性绑定或关闭时移除。
- **P3-29** `btnToggleSlotCheck` 静态文案 `时段检测: 开`（`913`）与默认 `false`（`4824`）不符，JS 加载前有错误闪现。建议初始 HTML 改 `时段检测: 关`。
- **P3-30** `btnEditRooms.onclick` 在 `13440` 与 `20055` 重复赋值（后者胜出），冗余，保留其一。

## 二、运行良好、无需改动的部分（经核对）

- **单人 assign/clear/renew 走真正的数据库 CAS**：`executeConditionalRoomCloudMutation` 用 `.eq('version', 期望值).select()`，零行返回即 `VERSION_MISMATCH`，回滚 + 从权威远端行重同步 + 调度对账。两位老师抢同一房间，第二个正确失败——这是对的范式。
- **转移预检**校验两房间版本与占用者、且有真实回滚路径。
- **可疑清空守卫**（远端 null 占用且版本 ≤ 本地则丢弃）防止乱序/过期清空覆盖更新的占用。
